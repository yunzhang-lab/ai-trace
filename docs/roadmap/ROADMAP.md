# 目标路线图

> 这不是纯时间维度的阶段说明，而是**从粗到细**、按对象层级推进的路线图。
> 每个目标都要执行一个完整的“公开-私有”反馈闭环：
> 1. 在公开仓（ai-trace-open）打磨并收敛规范。
> 2. 将确认更优的规范反哺到私有仓（~/.ai-trace），驱动私有目录、数据和流程的对齐。
> 3. H5 交互需求不独立滞后，而是随着规范的深入**同步生长**。

## 为什么是 5 类

| 类别 | 作用 | 推进逻辑 |
| --- | --- | --- |
| 结构类 | 先把公开仓、私有根、目录边界讲清楚 | 物理脚手架搭建 |
| 身份类 | 统一 Agent 卡片、状态、注册与权限 | 明确“谁在干活” |
| 对象类 | 统一偏好、教训、知识、项目、决策、冲突、审计、状态 | 明确“管理什么名词” |
| 流程类 | 统一原始日志、归并、审计、审批流、周报、回溯 | 明确“执行什么动词” |
| 协作基建类 | 输出标准 `SKILL.md`，挂载到各 Agent 入口 | 让 AI 真正接入跑起来 |

## 目标总表

| 类别 | 目标 | 公开规范要完善什么 | 私有仓要修改什么 | H5 需求 (跟随生长) | 完成标准 |
| --- | --- | --- | --- | --- | --- |
| 结构类 | 确认公开仓与私有根的边界 | `AI_TRACE_SYSTEM.md`、`INDEX.md`、`README.md`、公开/私有关系说明 | 按新规范重构私有根目录结构；删除废弃的顶层目录；导入 `process/` 规范 | 只要基础壳，先能看见主菜单和页面分区框架 | 目录结构、文档入口、边界说明三者一致 |
| 身份类 | 统一 Agent 卡片与注册 | `AGENT_SPEC.md`、`registry` 结构、Agent 类型分层、启动/退出动作 | 建立 Agent 卡片、状态文件、注册入口 | 展示 `Office` (Live Floor)，能看 Agent 卡片、状态和监控 | 任何 Agent 进来都能先知道自己是谁、能做什么、该看什么 |
| 对象类 | 统一内容对象模型 | `CONTENT_MODEL_SPEC.md`、`ALIGNMENT.md`、明确对象唯一 ID 和 `merged_from` 溯源链设计 | 清理私有仓的 `profile / work / governance / runtime` 落点数据 | 增加 `Wiki` 与 `Profile / Objects` 视图，实现对象的静态展示和多维度筛选器 | 对象类型和公私归属不再混淆，溯源链逻辑自洽 |
| 流程类 | 统一审批流、归并与审计 | `INFORMATION_FLOW_SPEC.md`、审批流、Audit 追加格式、H5 Action 触发机制、冲突判例沉淀 | 跑通一次真实流转：原始日志提炼 -> 生成 Action -> 追加 `audit` -> `approved` 落盘 | 增加 `Sync` 与 `Conflicts`，实现按钮点击生成动作指令，并接收执行回执 | 每条流程都能说清输入、输出、动作发起、执行落盘和审计回溯 |
| 协作基建类 | 多 Agent 统一技能挂载 | `SKILL.md` (协作手册) 与输出模板 | 部署统一 Skill；各 Agent 原生入口（如 `CLAUDE.md`）挂载该 Skill | 增加 `Skills`，支持一键复制 Agent 的接入指令 | 新 Agent 只要载入该 Skill，就能自动遵循边界和交接规范 |

## 执行顺序

| 顺序 | 类别 | 从粗到细的逻辑 |
| --- | --- | --- |
| 1 | 结构类 | **最粗的框**：先把地盘划分好，房子建起来。H5 有个大屏壳子。 |
| 2 | 身份类 | **工位入驻**：给员工（Agent）发工牌定权限。H5 能看到员工坐在工位上（Office）。 |
| 3 | 对象类 | **物料定义**：定义大家桌上的材料（偏好、知识）长什么样。H5 把这些材料在书架上摆好（Wiki/Profile）并能被检索。 |
| 4 | 流程类 | **业务流转**：材料怎么从这桌传到那桌（提炼、审批、冲突处理）。H5 提供审批章（Sync/Conflicts）并记录操作账本（Audit）。 |
| 5 | 协作基建类 | **自动化跑通**：把操作规程发给新员工（Skill）。H5 提供快速入职指南（Skills）。 |

## 收口

- 公开仓先按上表打磨。
- 每个目标完成后，再按对应规范回写私有仓。
- **H5 不再独立滞后**：推进到哪一层，H5 就把那一层的可视化和动作入口补齐。
- 目标不是一次性定稿，允许来回打磨，但每次只推进一个类。
