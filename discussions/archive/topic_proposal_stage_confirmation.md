# 阶段确认与解锁边界讨论稿

> updated_at: 2026-06-21
> status: approved
> participants: [Codex, Antigravity, Human]
> location: `discussions/topic_proposal_stage_confirmation.md`

本文档的目标不是继续扩实现，而是先确认：

1. 当前阶段到底怎么切
2. 每个阶段的正式目标是什么
3. 每个阶段允许解锁到什么精度
4. 哪些内容必须等进入相邻阶段后才能定义

当前建议：**在没有重新确认阶段之前，不继续扩更多阶段内 SPEC，也不继续推进超出当前阶段边界的实现。**

---

## 1. 为什么现在要先确认阶段

经过最近一轮收口后，我们已经形成了较稳定的三层文档职责：

- `README.md`
  负责系统总目标、主流程、公私边界与全局裁决
- `phases/ROADMAP.md`
  负责阶段推进顺序与阶段解锁规则
- `drafts/ALIGNMENT.md`
  负责跨阶段通用术语、状态、字段与动作原则

但当前仍有一个关键问题没有最后拍板：

> 我们虽然知道“近细远粗”，但还没有最终确认“每个阶段的边界到底停在哪里”。

如果这个问题不先确认，后面继续写 SPEC 或做实现时，很容易再次出现：

- 把后续阶段的内容提前写细
- 把目录、文件名、载体形式过早锁死
- H5 或后端能力在无意中超出当前阶段

---

## 2. 当前建议的阶段职责

下面不是最终结论，而是建议作为本轮讨论的基础版本。

### Phase 1：骨架收缩

目标：

- 收缩公开仓结构
- 明确公私边界
- 建立总纲、路线图、草稿区、讨论区的职责边界

应结束在：

- 6 根目录确认
- V2 总纲确认
- 路线图与对齐工作台职责分离

不应继续承担：

- 具体会话索引设计
- 具体私有仓目录命名
- 候选与审核的数据结构设计

### Phase 2：Scan / Register

目标：

- 扫描本地 Agent
- 生成候选
- 用户确认注册
- 建立 Agent Card 与映射关系

应结束在：

- `scan` 命令可输出候选
- H5 `Home / Agents` 可展示候选、注册表单、队列预览、已注册映射
- Phase 2 所需最小私有落点被定义

不应继续承担：

- 会话索引格式
- Bookmark 状态写入机制
- 候选提炼逻辑
- H5 会话详情与 Flow 审核页

### Phase 3：Index / Mark

目标：

- 对已授权 Agent 建最小会话索引
- 仅支持最小显式标记
- H5 只读展示最小会话视图

应结束在：

- `index` 命令可输出最小索引
- 第一版只提取最小会话元数据
- 第一版显式标签扫描可用
- H5 `Wiki / Flow` 最小只读视图可工作
- Mock / Real 数据切换链路跑通

不应继续承担：

- 工具子命令提取
- 文件列表提取
- H5 手工翻转 bookmark
- 本地 API 服务
- 多会话自动归并

### Phase 4：Extract / Review / Sync / Audit

目标：

- 将已标记会话提炼为候选
- 建立审核动作
- 建立同步与审计闭环

应结束在：

- Candidate 结构稳定
- 审核动作链路跑通
- Sync / Audit 私有落点被定义

不应继续承担：

- 稳定模板 Promote
- Skill 正式资产化

### Phase 5：Promote / Skill

目标：

- Promote 稳定模板
- 封装迁移工具
- 逐步形成 Skill 正式交付方式

应结束在：

- 稳定对象模板进入 `templates/`
- 迁移工具与正式 Skill 方案具备最小可用性

## 3. 当前建议的阶段确认规则

建议正式采用以下规则：

### 3.1 总纲只定义全局稳定边界

总纲允许长期保留：

- 主流程
- 公私边界
- 三层架构
- 数据载体原则
- 线性主干 + 受控分支 + 审计回流

总纲不提前定义：

- 私有仓固定目录名
- 阶段内具体文件格式
- 中远期对象模板细节

### 3.2 路线图决定何时解锁什么

路线图负责决定：

- 哪个阶段允许定义哪些目录
- 哪个阶段允许定义哪些文件格式
- 哪个阶段允许点亮哪些 H5 页面
- 哪个阶段允许增加哪些后端能力

### 3.3 阶段 SPEC 只写相邻阶段

建议只对：

- 当前阶段
- 下一阶段

编写详细 SPEC。

更远阶段只保留：

- 方向
- 依赖关系
- 不做什么

## 4. 本轮真正需要拍板的问题

这份讨论稿希望优先收口的，不是字段，而是阶段本身。建议用户重点判断以下问题：

1. **Phase 2 的正式边界是否只到 `Scan -> Register`？**
   即：候选、注册、映射、队列预览完成后，Phase 2 就结束。

2. **Phase 3 的正式边界是否只到 `Index -> Mark` 的最小只读闭环？**
   即：只做最小索引、最小显式标签扫描、最小只读视图，不进入写入式 H5 标记和复杂提炼。

3. **是否需要额外引入一个过渡小阶段（例如 Phase 3.5）？**
   例如用于承接：
   - H5 手工翻转 bookmark
   - 本地 API 服务
   - 更强的会话筛选

4. **是否确认“私有仓目录结构只在相邻阶段定义”这一规则？**
   即：
   - 总纲不预设目录
   - 路线图只决定哪个阶段允许定义
   - 真正目录结构在阶段 SPEC 里确定

5. **是否确认当前先不扩更多远期对象模型？**
   即：
   - Project / Knowledge / Preference / Governance 的最终模板
   - Skill 正式封装
   - 重型知识库/数据库方案
   暂时都只保留方向，不作为近期讨论重点

---

## 5. 当前建议结论

我当前的建议是：

1. 先把阶段本身重新确认
2. 再锁每个阶段的 DoD 与“不做什么”
3. 再写对应阶段的 SPEC
4. 最后才继续推进代码实现

这样可以避免：

- 文档继续扩但阶段未定
- 阶段边界再次漂移
- 目录和数据落点被过早锁死
